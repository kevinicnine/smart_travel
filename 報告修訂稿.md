## 摘要（建議替換）

本研究提出一套「基於 AI 景點分析之智慧旅遊規劃系統」，目標在大幅降低使用者於旅行前蒐集資訊與安排行程的時間成本。系統採用 Flutter 行動應用與 FastAPI 後端，整合 Google Maps 資料爬取之景點資訊，並以 TF‑IDF 結合 LinearSVC 進行景點分類，後續以語意向量檢索與權重化排序（考慮使用者旅遊目的、偏好與時間情境）自動生成個人化行程。系統同時支援路線最佳化（考量距離、營業時間與移動時間）與主動式推播機制，能在適當時機向使用者推薦行程並依回饋動態調整排列。實作上強調模組化與可擴展性，提供資料匯入、索引更新與模型部署流程，期望在提升推薦準確度的同時，顯著降低使用者的規劃成本與操作負擔。

### Abstract (English)

This study proposes an AI-driven travel planning system that analyzes points of interest (POIs) to help users quickly generate personalized itineraries and reduce the time cost of pre-trip research and planning. The system is implemented with a Flutter mobile application and a FastAPI backend, integrates POI data collected via Google Maps scraping, and uses TF‑IDF combined with LinearSVC for POI classification. Subsequent stages apply semantic vector retrieval and weighted ranking—taking into account user travel purpose, preferences, and temporal context—to automatically produce personalized itineraries. The system also supports route optimization (considering distance, opening hours, and travel time) and a proactive push mechanism that recommends itineraries and dynamically adjusts them based on user feedback. The implementation emphasizes modularity and scalability and provides workflows for data import, index updates, and model deployment, aiming to improve recommendation accuracy while significantly reducing users' planning effort and interaction burden.

## 前言

隨著人工智慧技術的快速發展，旅遊資訊的產生與流通日益增多，但使用者在旅行前仍需花費大量時間蒐集景點資料、比較行程並手動安排日程。本研究旨在開發一套基於 AI 景點分析的智慧旅遊規劃系統，透過自動化的資料蒐集與處理、語意分類與向量檢索，結合使用者偏好與時間情境，主動提供個人化且動態調整的行程建議。本文首先說明系統設計與核心演算法，接著描述資料管線與實作細節，並呈現實驗評估與部署流程，最後討論系統限制與未來改進方向。

3. 製作結果：系統已完成端到端流程的實作與測試：包含景點資料匯入與預處理、分類模型部署、向量索引建立、行程生成功能、路線最佳化以及 LINE 推播整合。使用者可透過介面設定偏好、查詢推薦並取得可編輯的行程建議；在即時情境變化（如天氣或交通異常）時，系統能自動重新排序或替換景點並主動通知使用者。初步測試顯示系統能有效縮短使用者規劃時間並提高推薦之個人化程度，具備實務應用價值與後續擴展潛力。

## 製作理論探討

本章節針對本系統所依據之核心理論與演算法進行分段介紹，說明各模組的設計動機與技術細節，便於在報告中作為理論依據與實作說明引用。

1. 智慧旅遊推薦概念

 - 目標：結合使用者偏好、旅遊目的與即時情境，從大量景點中自動挑選並排列出符合需求的行程，減少使用者手動搜尋與決策成本。
 - 方法：以召回（recall）+排序（ranking）之雙階段推薦流程為基礎——先由向量檢索與類別篩選召回候選景點，再以加權排序整合偏好、時間/距離限制與情境因素產生最終行程。

2. 景點分類與特徵工程

 (一) 資料蒐集與前處理

 - 資料來源：使用 Google Places API 與自建爬蟲收集景點名稱、描述、經緯度、營業時間、評分等欄位，並將原始文字做清洗（去除特殊字元、HTML 標記、統一編碼）。
 - 欄位整理：對描述文字進行中文斷詞、停用詞移除、詞形正規化，並保留地理與時間相關欄位以供後續路線與情境分析使用。

 (二) 特徵轉換與分類模型

 - 特徵轉換：以 TF‑IDF 將景點文字（名稱、描述、類別標籤）轉換為向量特徵，以捕捉詞頻與重要性。必要時加入詞向量或 sentence embedding 作為補強特徵。
 - 分類模型：採用 LinearSVC 作為分類基礎模型，將景點標注為多類型（如美食、自然、文化、購物等），模型輸入為 TF‑IDF 特徵與其他結構化欄位（如評分、人氣）。LinearSVC 在高維稀疏特徵上的穩定性與效率使其成為合適選擇。
 - 關鍵字判斷：透過重要性分數（TF‑IDF weight）判別關鍵描述字詞，作為分類解釋與進一步的召回規則依據。

3. 向量檢索與召回策略

 - Embedding 生成：將景點文字或使用者查詢轉為語意向量（embedding），可使用現成中文/多語言 embedding 模型或微調模型以提高本地語意表現。
 - 向量索引：使用 FAISS/Milvus/Weaviate 等向量庫儲存 embdding，以支援高效的近鄰搜尋（ANN）召回。
 - 召回策略：結合類別過濾（由分類模組提供）、地理半徑過濾與向量相似度召回，確保候選集兼具相關性與地域可行性。

4. 情境感知（Context‑Aware）設計

 - 目的：將即時情境（天氣、交通、時間、節慶）納入推薦考量，使生成行程更符合當下可行性與使用者需求。
 - 實作：建立一個情境權重引擎，將外部情境資料映射為影響係數（例如雨天降低戶外景點權重、夜間不推薦日間營業景點），並在排序函數中以乘權或懲罰方式調整候選景點分數。

5. 行程排序與路線最佳化

 - 排序模型：最終排序考量包含使用者偏好分數、景點評價、向量相似度、地理順序相容性與情境權重，採用線性加權或學習排序 (learning-to-rank) 方法實作。
 - 路線優化：在同一天行程內使用旅行推銷員問題（TSP）近似或啟發式演算法（如 2‑opt, greedy）結合營業時間與移動時間限制生成可行路線，以降低移動成本與時間浪費。

6. 主動推播與使用者回饋迴路

 - 主動推播：系統可根據預設條件或模型判斷在合適時間推送行程建議（例如出發前數日、天氣變化時等），LINE 機器人作為通知入口，並提供快速接受/拒絕/微調選項。
 - 回饋機制：使用者對推播或建議的互動（點擊、接受、修改）將回傳給後端作為行為資料，進行日後排序權重調整或再訓練資料累積，形成持續學習迴路。

## 軟體程式分析

本節分段說明系統的軟體設計、主要模組、核心檔案位置、API 與資料流，並提供實作細節與維運考量，方便將技術細節放入報告或附錄中。

1. 系統整體架構（Software Architecture）

 - 採分層模組化架構：前端（Flutter 行動應用）、後端 API（FastAPI）、資料層（Postgres、向量索引）、模型服務（embedding 與分類）、與 ETL/管理工具。
 - 通訊方式：前端與後端透過 HTTP/HTTPS REST API 互動；後端與模型/向量庫可透過 gRPC/HTTP 或客製化 client SDK 串接；管理任務（匯入、重建索引）以 CLI 或管理 API 觸發。

2. 技術棧（Tech Stack）

 - 前端：Flutter（Dart）用於行動端示範介面，負責查詢輸入、結果顯示與行程編輯。
 - 後端：FastAPI（Python）提供 RESTful API，負責請求路由、業務邏輯整合、認證與日誌。
 - ML/向量：TF‑IDF + LinearSVC（分類），embedding 使用現成語言模型（或 sentence-transformers）；向量儲存採 FAISS/Milvus/Weaviate。
 - 資料庫：Postgres（結構化資料）、物件儲存（如 S3）儲存大型資源。
 - 通訊/通知：LINE Messaging API（推播）、第三方天氣/交通 API 作為情境來源。

3. 專案結構與主要檔案（Project Structure）

 - `smart-travel/app/etl/`：資料爬蟲、清洗、映射與匯入腳本（如 `import_data.py`、`normalize.py`）。
 - `smart-travel-api/app/main.py`：FastAPI 啟動入口與設定。
 - `smart-travel-api/app/routers/`：主要路由模組，例如 `keywords.py`（關鍵字查詢）、`recommend.py`（推薦）、`admin.py`（管理/匯入）。
 - `smart-travel-api/app/models.py`：ORM 與資料結構定義（SQLAlchemy / Pydantic schema）。
 - `smart-travel-api/app/services/`：包含分類服務、embedding 產生器、向量檢索封裝、路線最佳化模組與情境引擎。
 - `smart-travel-frontend/`：Flutter 程式碼，分為 UI、state management 與 API client。

4. 後端模組細項（Backend Components）

 - 路由層（Routers）：驗證輸入、錯誤處理、回傳統一格式（status, data, error）。
 - 服務層（Services）：實作業務邏輯（召回、排序、路線優化、推播排程），將外部依賴（DB、向量庫、外部 API）抽象包裝，方便測試與替換實作。
 - 資料層（Repositories）：對 DB 的 CRUD 操作集中管理，並提供事務處理界面。
 - 背景任務（Workers）：使用 Celery / RQ 或 FastAPI-native background tasks 處理資料匯入、索引重建與排程推播。

5. 前端模組細項（Frontend Components）

 - 查詢頁面：收集使用者輸入（關鍵字、偏好、天數），呼叫 `/api/keywords/query` 或 `/api/recommend/itinerary`。
 - 結果頁面：以卡片 / 地圖呈現景點，提供接受/替換/編輯行程的快速操作。
 - 行程編輯器：允許拖放、儲存與導出行程，並觸發後端的路線最佳化 API。

6. API 設計範例（Endpoints & Payloads）

 - `POST /api/keywords/query` — body: `{"query":"...","limit":10}` → 回傳候選景點陣列。
 - `POST /api/recommend/itinerary` — body: `{"user_preferences":{},"seed_spots":[]}` → 回傳結構化行程（日期、時段、景點序列）。
 - `POST /api/admin/import` — 管理端觸發匯入/索引重建（需認證）。

7. 資料庫與索引管理（DB & Indexing）

 - Schema 建議：`spots`（id, name, lat, lon, category, desc, open_hours, score）、`itineraries`（user_id, date, items）、`embeddings`（spot_id, vector_ref）。
 - 索引管理：定期或在資料變更時觸發向量索引重建；對大型資料集採增量更新以降低重建成本。

8. 設定、祕密管理與部署配置（Config & Secrets）

 - 所有敏感資訊請置於環境變數或秘密管理系統（`.env`、Vault），不要硬寫於程式碼。
 - 提供 `docker-compose.yml` 與 `.env.example`，列出必要變數（`DATABASE_URL`, `VECTOR_DB_URL`, `LINE_CHANNEL_TOKEN` 等）。

9. 日誌、監控與測試（Logging, Monitoring, Testing）

 - 日誌：後端採 structured logging（JSON）輸出，紀錄 request id、latency、錯誤堆疊以利追蹤。
 - 監控：建議導入 Prometheus + Grafana 監控 API 延遲、錯誤率與任務隊列長度。
 - 測試：單元測試（服務層、路由層）、整合測試（API endpoints、資料匯入）、端到端測試（前端呼叫後端）。CI 可使用 GitHub Actions 進行 lint、測試與 build。

10. 安全性與權限（Security & Auth）

 - 管理 API 加強認證（JWT / OAuth2），對敏感管理動作（匯入、索引重建）額外驗證。
 - 對外 API 請求限流與輸入驗證，防止惡意或大量請求造成服務中斷。

11. 性能與擴展性（Performance & Scalability）

 - 向量檢索採 ANN（FAISS/Milvus）以支援大規模相似度搜尋；模型服務可採 batch 處理以提高 throughput。
 - 採用水平擴展（multiple API replicas + load balancer）、資料庫讀寫分離與快取（Redis）以降低延遲。

12. 開發流程與版本管理（Dev Workflow）

 - 建議採 GitFlow 或 trunk-based workflow，PR 模式進行程式碼審查，並在合併前透過 CI 執行測試。
 - 文件化 API（OpenAPI / Swagger）、部署步驟與常用 CLI 指令以利維運。

 - 功能測試：單元測試與整合測試（驗證 API 回傳、ETL 輸入結果、分類模型輸出格式）。

## 製作理論探討

本章節針對本系統所依據之核心理論與演算法進行分段介紹，說明各模組的設計動機與技術細節，便於在報告中作為理論依據與實作說明引用。

1. 智慧旅遊推薦概念

 - 目標：結合使用者偏好、旅遊目的與即時情境，從大量景點中自動挑選並排列出符合需求的行程，減少使用者手動搜尋與決策成本。
 - 方法：以召回（recall）+排序（ranking）之雙階段推薦流程為基礎——先由向量檢索與類別篩選召回候選景點，再以加權排序整合偏好、時間/距離限制與情境因素產生最終行程。

2. 景點分類與特徵工程

 (一) 資料蒐集與前處理

 - 資料來源：使用 Google Places API 與自建爬蟲收集景點名稱、描述、經緯度、營業時間、評分等欄位，並將原始文字做清洗（去除特殊字元、HTML 標記、統一編碼）。
 - 欄位整理：對描述文字進行中文斷詞、停用詞移除、詞形正規化，並保留地理與時間相關欄位以供後續路線與情境分析使用。

 (二) 特徵轉換與分類模型

 - 特徵轉換：以 TF‑IDF 將景點文字（名稱、描述、類別標籤）轉換為向量特徵，以捕捉詞頻與重要性。必要時加入詞向量或 sentence embedding 作為補強特徵。
 - 分類模型：採用 LinearSVC 作為分類基礎模型，將景點標注為多類型（如美食、自然、文化、購物等），模型輸入為 TF‑IDF 特徵與其他結構化欄位（如評分、人氣）。LinearSVC 在高維稀疏特徵上的穩定性與效率使其成為合適選擇。
 - 關鍵字判斷：透過重要性分數（TF‑IDF weight）判別關鍵描述字詞，作為分類解釋與進一步的召回規則依據。

3. 向量檢索與召回策略

 - Embedding 生成：將景點文字或使用者查詢轉為語意向量（embedding），可使用現成中文/多語言 embedding 模型或微調模型以提高本地語意表現。
 - 向量索引：使用 FAISS/Milvus/Weaviate 等向量庫儲存 embdding，以支援高效的近鄰搜尋（ANN）召回。
 - 召回策略：結合類別過濾（由分類模組提供）、地理半徑過濾與向量相似度召回，確保候選集兼具相關性與地域可行性。

4. 情境感知（Context‑Aware）設計

 - 目的：將即時情境（天氣、交通、時間、節慶）納入推薦考量，使生成行程更符合當下可行性與使用者需求。
 - 實作：建立一個情境權重引擎，將外部情境資料映射為影響係數（例如雨天降低戶外景點權重、夜間不推薦日間營業景點），並在排序函數中以乘權或懲罰方式調整候選景點分數。

5. 行程排序與路線最佳化

 - 排序模型：最終排序考量包含使用者偏好分數、景點評價、向量相似度、地理順序相容性與情境權重，採用線性加權或學習排序 (learning-to-rank) 方法實作。
 - 路線優化：在同一天行程內使用旅行推銷員問題（TSP）近似或啟發式演算法（如 2‑opt, greedy）結合營業時間與移動時間限制生成可行路線，以降低移動成本與時間浪費。

6. 主動推播與使用者回饋迴路

 - 主動推播：系統可根據預設條件或模型判斷在合適時間推送行程建議（例如出發前數日、天氣變化時等），LINE 機器人作為通知入口，並提供快速接受/拒絕/微調選項。
 - 回饋機制：使用者對推播或建議的互動（點擊、接受、修改）將回傳給後端作為行為資料，進行日後排序權重調整或再訓練資料累積，形成持續學習迴路。

本系統以前端 Flutter 行動應用程式搭配 FastAPI 後端架構進行開發，並整合 Google Maps API 透過
景點資料爬蟲蒐集旅遊資訊，利用 TF-IDF 與 LinearSVC 模型進行景點分類，同時結合使用者旅遊目的、
偏好權重及時間感知排序機制，自動生成符合使用者需求之旅遊行程。此外，系統亦可依據景點距離、
營業時間與移動時間進行路線最佳化，提高旅遊安排效率與便利性。
